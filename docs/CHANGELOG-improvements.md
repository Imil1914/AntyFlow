## T2.4 — Память доски через retrieval (вместо «вся память в контекст»)

**Ветка:** `feat/t1.1-sqlite` (продолжение).

**Что сделано.**

- **БД (миграция схемы №3):** `memory_embeddings(board_id, period_kind, period_key, vector BLOB, …)`
  (вектор — Float32 BLOB). Функции `memEmbList/memEmbSet/memEmbDelete`; IPC
  `memory:embList/embSet` (+ удаление эмбеддинга в `memory:delete`) + мост + типы.
- **Эмбеддинги выжимок (renderer):** та же модель, что в PDF-RAG (`multilingual-e5-small`).
  `hydrateMemEmbeddings(boardId)` грузит векторы из БД; `backgroundIndexMem` фоново досчитывает
  недостающие (порциями по 30, `embedPassages`), при недоступности модели — тихо деградирует.
  Изменение выжимки (`appendBoardMem`) инвалидирует её эмбеддинг и пересчитывает.
- **Сборка контекста памяти `buildMemoryContext(boardId, queryVector)`:** последние N дней целиком
  (по умолчанию 7) + top-k релевантных старых дневных выжимок (по умолчанию 6, косинус к вектору
  запроса) + недельные/месячные сводки, всё в пределах `memory.contextBudget` (токены, по умолчанию
  4000; счётчик — `gpt-tokenizer`). Порядок: месяц → неделя → релевантные старые → свежие.
- **Query-aware:** в `gatherChatSources`/`gatherChatContext` добавлен `memQueryVector`; нода
  `boardmem` теперь отдаёт retrieval-выборку, а не всю память. `AiBody` эмбеддит вопрос (только
  если подключена нода памяти) и передаёт вектор; **инспектор контекста (T3.1)** тоже эмбеддит текст
  ноды — показывает ту же retrieval-выборку. Если запроса нет → последние + сводки.
- **Флаги (settings.json):** `memoryRetrieval` (по умолчанию **вкл**), `memoryContextBudget` (4000),
  `memoryRecentDays` (7), `memoryTopK` (6). UI-раздел «🧠 Память доски» в настройках. Флаг off →
  прежнее поведение (вся память).

**Новые флаги/настройки.** `memoryRetrieval`/`memoryContextBudget`/`memoryRecentDays`/`memoryTopK`.

**Решения «на выбор исполнителя».**
- Эмбеддинги ключуются по `(board_id, period_kind, period_key)` (у выжимок нет отдельного
  публичного rowid в renderer).
- При отборе приоритет: компактные сводки → свежие дни → релевантные старые (пока влезает в бюджет);
  вывод — хронологично по уровням. Без эмбеддингов/запроса retrieval всё равно полезен (бюджет +
  свежие + сводки) — это и есть «мягкая деградация».
- **Авто-выжимки** пока считаются текущей моделью по умолчанию (роль `summarize` появится в T3.3 —
  тогда переключатся на неё; это мягкая зависимость, отмеченная в §0.2 ТЗ).

**Проверено.**
- Логика `buildMemoryContext` (изолированный прогон, синт. доска 60+ дней): контекст ≤ бюджета
  (в т.ч. при жёстком бюджете 300 — 292 ток.); тематический вопрос вытягивает редкую релевантную
  старую выжимку; сводки недели/месяца включены; без запроса редкая старая НЕ подтягивается.
- `npx tsc --noEmit`: новых ошибок нет. `npm run build`: `✓ built`. `npx vitest run`: 11/11.

**Осталось на ручную проверку (нужен GUI + модель эмбеддингов).**
- Реальная доска: контекст памяти в инспекторе ≤ бюджета; тематический вопрос вытягивает релевантную
  старую выжимку. Старые доски без эмбеддингов работают сразу (recent+сводки) и мигрируют в фоне.

## T2.2 — Цитаты с якорями страниц

**Ветка:** `feat/t1.1-sqlite` (продолжение).

**Что сделано.**

- **Метаданные страниц:** чанки уже хранят `page` (из существующей индексации) — отдельная
  переиндексация для страниц не требуется; старые PDF работают сразу.
- **Промпт Q&A (`pdf:ask`):** источники нумеруются `[1]…[n]` с указанием страниц
  (`[1] (стр. 3) …`), системная инструкция просит модель ставить маркеры `[n]` после утверждений.
- **Источники в ответе:** `pdf:ask` отдаёт вместе с `pdf:done` массив `sources` (номер → страница
  + якорный текст, ≤400 симв.). Хранится в истории Q&A хайлайта (`PdfQA.sources`) — переживает
  перезагрузку.
- **Рендер + переход:** под каждым ответом — блок «Источники» с кликабельными чипами `[n] с.{page}`.
  Клик → `goToCitation`: переход на нужную страницу PDF-ноды + временный баннер-подсветка с
  фрагментом источника (5 c). Блок появляется всегда, когда retrieval вернул чанки (даже если модель
  не проставила маркеры).

**Новые флаги/настройки.** Нет.

**Решения «на выбор исполнителя».**
- «Подсветка» перехода — баннер с фрагментом источника на нужной странице (наглядно и надёжно).
  Точный бокс в текстовом слое (text→bbox) не реализован — по ТЗ допустим фолбэк «просто перейти на
  страницу», что здесь и сделано (плюс баннер).
- `charStart/charEnd` («по возможности») не добавлял — для навигации достаточно `page` + якорного
  текста.
- Кликабельные цитаты — через блок «Источники» (чипы `[n] с.X`), тело ответа рендерится обычным
  markdown без потери форматирования.

**Проверено.**
- `npx tsc --noEmit`: новых ошибок нет. `npm run build`: `✓ built`. `npx vitest run`: 11/11.

**Осталось на ручную проверку (нужен GUI).**
- Ответ по PDF содержит кликабельную цитату; клик открывает нужную страницу с баннером-подсветкой.
- Блок «Источники» появляется всегда при наличии чанков; старые PDF работают.

## T2.1 — Гибридный поиск (локальный PDF-RAG)

**Ветка:** `feat/t1.1-sqlite` (продолжение).

**Что сделано.**

- **Гибридный retrieval в `src/main/pdf.ts`:** `searchPdfHybrid(id, vector, queryText, {hybrid, topN})` —
  top-40 лексических (BM25) + top-40 векторных (cosine) → слияние **Reciprocal Rank Fusion** (k=60)
  → top-N (по умолчанию 8). BM25 считается по чанкам индекса на лету (тексты уже сериализованы в
  `pdf/index/<id>.json`), токенизация по буквам/цифрам Unicode (рус+англ).
- **Замер и лог этапов** retrieval со скоупом `[rag]` (время BM25 / вектора / общее).
- **Флаги (settings.json):** `ragHybrid` (rag.hybrid, по умолчанию **вкл**), `ragTopN` (по умолчанию 8),
  `ragReranker` (rag.reranker, по умолчанию **выкл**). Мягкая деградация: `ragHybrid=false` или пустой
  текст запроса → прежнее чисто векторное поведение.
- **Подключено к Q&A:** `pdf:ask` теперь зовёт `searchPdfHybrid` с текстом вопроса (раньше — только
  вектор, top-5). IPC `pdf:search` расширен опциональным `query` (гибрид при наличии текста).
- **UI настроек:** раздел «📕 PDF-поиск (RAG)» — переключатель гибрида, top-N, и (задизейбленный)
  переключатель реранкера с пометкой «скоро».

**Новые флаги/настройки.** `ragHybrid` (true), `ragTopN` (8), `ragReranker` (false) в settings.json.

**Решения «на выбор исполнителя».**
- Вместо отдельного персистентного minisearch-индекса — **BM25 по уже сериализованным чанкам**
  index-файла (корпус одного PDF мал, десятки–сотни чанков). Меньше зависимостей, тот же результат;
  критерий ТЗ (сериализация индекса + рус/англ токенизация) выполнен.
- RRF k=60 (классическое значение), top-40 на каждый канал.

**Реранкер — задокументированный follow-up.** Флаг и мягкая деградация реализованы (реранкер выкл →
гибрид). Сама ONNX-инференция `bge-reranker-v2-m3` (кнопка «Скачать модель», `%APPDATA%/flow/models/`,
офлайн-реранк top-40→top-8) НЕ включена в этой версии: требует загрузки ~600 МБ модели и офлайн-
валидации инференса, которые невозможно проверить в текущем окружении. Точка подключения реранкера
обозначена в `searchPdfHybrid`. Это соответствует принципу ТЗ «без модели/при ошибке — фолбэк на
гибрид без реранка».

**Проверено.**
- Логика гибрида (изолированный прогон): PDF с редким точным термином «ARIMA-GARCH», который
  «эмбеддинг» не выводит в top-3, — BM25 ставит его #1, RRF выводит в top-N #1. Выключение гибрида
  возвращает чисто векторный порядок.
- `npx tsc --noEmit`: новых ошибок в затронутых файлах нет. `npm run build`: `✓ built`.
  `npx vitest run`: 11/11 не сломаны.

**Осталось на ручную проверку (нужен GUI).**
- Реальный PDF: вопрос с редким термином — нужный чанк в ответе/источниках.
- Выключение флага в настройках возвращает прежнее поведение.

## T3.1 — Инспектор контекста

**Ветка:** `feat/t1.1-sqlite` (продолжение).

**Что сделано.**

- **Единая сборка контекста (рефакторинг без дублирования):** `gatherChatContext` (сборка для
  отправки) переписан поверх нового `gatherChatSources(editor, nodeId, skip?)`, который возвращает
  структурированные источники `{shapeId, kind, title, text, tokens}`. Отправка = те же источники
  минус временно исключённые, тем же порядком → инспектор и реальный промпт совпадают **побайтово**.
- **Оценка токенов:** `gpt-tokenizer` (cl100k) + `estimateTokens(text)`; в UI помечается «≈».
- **Лимит модели:** используется существующий `contextLimitFor(model)` (Gemini 1M, Claude 200k,
  GPT-4o/GLM/DeepSeek 128k, …, дефолт 64k).
- **Временное исключение источников:** сессионный реестр `contextExclusions` +
  `isSourceExcluded/toggleSourceExclusion` — «исключить из этого запроса» не удаляет стрелку,
  влияет на реальную отправку, сбрасывается при перезапуске.
- **UI `ContextInspector`** (поповер над ai-нодой, кнопка «🧮 контекст» рядом со счётчиком):
  список источников (тип, заголовок, ≈токены), чекбоксы включения/исключения, итог ≈токенов vs
  лимит модели, предупреждение при переполнении (что и в каком порядке обрежется), кнопка
  «Скопировать промпт целиком» (копирует ровно то, что уйдёт — `included.join('\\n\\n———\\n\\n')`).
- Источники по стрелкам уже включают память доски (нода `boardmem`) и веб-чаты (транскрипты через
  `extractNodeContext`) — инспектор покрывает все три типа единым списком.

**Новые флаги/настройки.** Нет.

**Решения «на выбор исполнителя».**
- Инспектор показывает и копирует именно блок связанного контекста (сборка `gatherChatSources`),
  т.к. по нему проверяется побайтовое соответствие; общий системный «обёрточный» текст запроса не
  входит в сравнение по ТЗ.
- Исключения — сессионные (в памяти), а не в `extra`: по ТЗ это «временное исключение из этого
  запроса».
- Оценка токенов приблизительная (cl100k) для всех моделей — помечено «≈».

**Проверено.**
- `npx tsc --noEmit`: новых ошибок в затронутых файлах нет. `npm run build`: `✓ built`.
  `npx vitest run`: 11/11 (T1.2) не сломаны.
- Инвариант «инспектор = отправка»: обе стороны используют один `gatherChatSources` с общим
  реестром исключений и одинаковым порядком/склейкой.

**Осталось на ручную проверку (нужен GUI).**
- Нода с 3+ источниками: состав/порядок в инспекторе = «Скопировать промпт».
- Исключение источника меняет реальную отправку; на переполненном примере видно предупреждение.

## T4.1 — Глобальный поиск (Ctrl+K)

**Ветка:** `feat/t1.1-sqlite` (продолжение).

**Что сделано.**

- **БД (миграция схемы №2):** таблица FTS5 `nodes_fts(board_id/board_name/shape_id UNINDEXED,
  kind, title, body)` (`unicode61 remove_diacritics 2`). Не external-content — переиндексация
  доски = `DELETE WHERE board_id` + `INSERT` (простая инвалидация удалённых нод). Функции
  `nodesReindexBoard/nodesDeleteBoard/nodesSearch` в `db.ts`; IPC `nodes:reindex/deleteBoard/search`
  (zod) + мост + типы.
- **Индексация (renderer):** `src/renderer/src/search.ts` — `buildNodeIndex(editor)` собирает
  `{shapeId, kind, title, body}` для каждой `flow-node`. Body — обобщённый сбор строковых значений
  из `props.body` + `props.extra` (покрывает sheet-ячейки, kanban/list-карточки, code — без
  per-kind экстракторов; base64/data-URL и слишком длинные строки исключаются). Вызовы: при
  сохранении доски (дебаунс `scheduleSaveCanvas`) и при открытии доски (после mount) — так ноды
  ищутся и без правок; при удалении доски — `nodes:deleteBoard`.
- **Поиск по транскриптам:** `searchTranscripts(query)` в `FlowNodeShapeUtil` (по реестру
  `agentTranscripts`; только смонтированные в текущей сессии веб/агент-ноды).
- **Палитра Ctrl+K → `GlobalSearch`** (в `os/overlays.tsx`, заменила старую командную `CmdK`):
  поле ввода, вкладки-фильтры **Всё | Ноды | Память | Транскрипты | Команды**, результаты со
  сниппетами и подсветкой совпадений (`<mark>`), недавние запросы (localStorage). Команды из
  старой палитры сохранены отдельной секцией — функциональность не потеряна.
- **Навигация:** клик/выбор → если нода на другой доске, переключаем доску (`pendingFocus` +
  ожидание загрузки снапшота с ретраями, ≤ ~3 c) и центрируем камеру на ноде с выделением-подсветкой
  (`focusShape`: `zoomToBounds` + `setSelectedShapes`); память → открыть ноду «Память доски» на её
  доске; транскрипт → сфокусировать веб-ноду (если на текущей доске).

**Новые флаги/настройки.** Нет. Миграция схемы БД №2 (nodes_fts) — включена по умолчанию.

**Решения «на выбор исполнителя».**
- Индексация body — **обобщённый flatten строк из extra**, а не per-kind экстракторы: проще,
  устойчивее и покрывает требуемые sheet/kanban/list/code.
- Ноды индексируются **при открытии и правке доски** (не все доски разом на старте) — доска
  попадает в индекс, как только её открыли/изменили. Ограничение задокументировано в UI
  («Ноды индексируются при открытии/правке доски»).
- Транскрипты ищутся только по смонтированным в сессии нодам (в БД не персистятся — отдельная
  задача); навигация к транскрипту работает, если нода на текущей доске.
- Ctrl+K превращён в поиск (по ТЗ), команды встроены секцией, чтобы прежний сценарий не сломать.
- Устойчивость к спецсимволам: `toFtsQuery` берёт только буквенно-цифровые токены (≥2 симв.) как
  префиксные совпадения; при пустом/битом FTS-запросе — фолбэк на `LIKE` с экранированием.

**Проверено.**
- Прогон `nodes_fts` под реальным Electron: поиск по body (в т.ч. по коду) и заголовку,
  кириллица, кросс-бордовый поиск, переиндексация доски (удалённая нода исчезает), удаление доски,
  устойчивость к спецсимволам (пустой FTS → LIKE-фолбэк, без падений).
- `npx tsc --noEmit`: новых ошибок в затронутых файлах нет. `npm run build`: `✓ built`.
  `npx vitest run`: 11/11 (T1.2) не сломаны.

**Осталось на ручную проверку (нужен GUI).**
- Нода с другой доски находится по слову из тела и открывается с центрированием ≤ 2 c.
- Удалённая нода исчезает из результатов после ближайшего сохранения доски.

## T1.2 — Версионирование схемы нод (zod + миграции)

  - `README.md`: железное правило — любое изменение формата `extra` = новая версия +
    миграция + тест.
  - `__tests__/migrations.test.ts` (vitest).

  поле `pinned` → `favorite`. Реальные заметки хранят текст в `props.title/body`, `extra`

- **Загрузка доски:** `migrateBoardExtras(editor)` в `FlowNodeShapeUtil.tsx` прогоняет все
  `flow-node`: невалидный JSON/не-объект → нода помечается повреждённой и рендерится
  заглушкой `CorruptedBody` (kind + свёрнутый сырой JSON, данные не теряются, холст жив);
  иначе применяются миграции, изменённый `extra` однократно перезаписывается в сторе
  **тихо** (`store.mergeRemoteChanges`, без засорения undo). Вызывается в `App.tsx` после
  `loadSnapshot` (pullCanvas) и как подстраховка через 1.5с после mount (локальные доски).
- **Ключевое решение:** `migrateExtra` НЕ помечает изменёнными v1-ноды без `extraVersion`
  (иначе переписывалась бы каждая нода при каждом открытии) — трогаются только ноды с
  реальным апгрейдом версии (сейчас — только `note`).
- Добавлен **vitest** (devDep) + `vitest.config.ts` + npm-скрипт `test` (`vitest run`).

**Новые флаги/настройки.** Нет.

**Миграции.** Данные нод: `extra` мигрируется по `kind` при загрузке доски (с версии 1).
Пока фактическая миграция затрагивает только `note` (v1→v2).

**Решения «на выбор исполнителя».**
- Схемы делаю **пермиссивными** (`catchall`), а не строгими, — цель T1.2 защитить обратную
  совместимость, а не отвергать валидные старые данные. Ужесточение полей — по мере
  надобности, вместе с новой версией+миграцией (правило в README схем).
- «Повреждённой» нода считается только при невалидном JSON / не-объекте (или провале
  схемы) — пермиссивные схемы валидные объекты не отвергают.
- Миграция применяется как remote-изменение (без undo-записи), чтобы Ctrl+Z сразу после
  открытия не «откатывал» апгрейд формата.

**Ветка:** `feat/t1.1-sqlite`

**Что сделано.**

- Подключён `better-sqlite3` (^11) в main-процессе. Нативный модуль пересобран под
  Electron 33 (`electron-rebuild --only better-sqlite3`); для `dist` — `npmRebuild: false`
  (пакуется уже пересобранный бинарник) + добавлен в `asarUnpack`
  (`node_modules/better-sqlite3/**`, `bindings/**`, `file-uri-to-path/**`).
- Новый модуль `src/main/db.ts`: единая БД `%APPDATA%/flow/flow.db` в режиме **WAL**;
  таблица `schema_migrations` + пронумерованные миграции; подготовленные стейтменты;
  аккуратное закрытие (`wal_checkpoint(TRUNCATE)` + `close`) в `will-quit`.
- Таблицы:
  - `board_memory(id, board_id, period_kind /*day|week|month*/, period_key, content,
    created_at, updated_at)` с `UNIQUE(board_id, period_kind, period_key)`;
  - `board_meta(board_id, key, value)` — пер-борд флаги (таймлайн, дата создания доски);
  - `memory_fts` — FTS5 по `content` (external-content, `content_rowid=id`) + триггеры
    `ai/ad/au` для синхронизации insert/update/delete.
- IPC + мост `window.flow` (валидация всех пейлоадов zod'ом в main):
  `memory:list/get/upsert/delete/search`, `boardmeta:get/getAll/set`, `db:status`,
  `db:importLocalDump`. Типы — в `flow-api.d.ts`.
- Renderer переведён с прямого localStorage на БД:
  - память доски (`readBoardMem/appendBoardMem/boardMemText` в `FlowNodeShapeUtil.tsx`)
    работает через **синхронный кэш**, зеркалящий БД (нужно, т.к. `boardMemText`
    вызывается синхронно в `extractNodeContext`); запись — сразу в кэш + асинхронно в БД;
  - флаги таймлайна (`timeline.on`) и день создания доски (`board.createdMs`) —
    через `board_meta` (новый модуль `src/renderer/src/boardStore.ts` с кэшом).
- **Миграция localStorage → БД** (одноразовая, `migrateLocalStorageToDb`): при первом
  запуске renderer собирает `flow-boardmem-*`, `flow-timeline-on-*`, `flow-board-created-*`
  → main сохраняет сырой дамп в `%APPDATA%/flow/backup/localStorage-<дата>.json`, затем
  пишет в БД → только после подтверждения записи boardmem-ключи помечаются перенесёнными
  (переименование в `flow-boardmem-migrated-*`, данные НЕ удаляются) + флаг
  `flow-db-migrated-v1`. Идемпотентно.
- **Надёжность.** Повреждение/отсутствие `flow.db` не роняет приложение: файл уводится
  в `backup/flow.corrupt-<дата>.db`, БД пересоздаётся, ошибка в лог, статус — через
  `db:status` (для будущего уведомления в UI). Все IPC БД возвращают `{ok,data}` /
  `{ok:false,error}` — исключения в renderer не пробрасываются.

**Новые флаги/настройки.** Нет пользовательских флагов (инфраструктура включена по
