# ТЗ: пакет улучшений AntyFlow (v1)

Документ для работы через Claude Code в репозитории `github.com/Imil1914/AntyFlow`.
26 задач в 5 эпиках. Каждая задача самодостаточна: цель → объём → критерии приёмки → файлы. Работать строго по одной задаче за сессию.

---

## 0. Контекст проекта (обязательно к прочтению перед любой задачей)

AntyFlow — локальное десктоп-приложение (Electron + React + tldraw): бесконечный AI-холст с нодами, мульти-агентным оркестратором, RAG и таймлайн-памятью досок.

Ключевые точки кодовой базы:

- `src/main/` — Electron main: IPC, провайдеры моделей (единая точка `callModel`, формат `providerId::modelId`, ретраи на 429/5xx), агенты, оркестратор (брокер + worker_threads), PDF, vault, AnythingLLM.
- `src/preload/index.ts` — мост `window.flow`; типы моста — `src/renderer/src/flowapi.d.ts`.
- `src/renderer/src/App.tsx` — холст, доски, сайдбар, таймлайн, диалоги.
- `src/renderer/src/shapes/FlowNodeShapeUtil.tsx` — единый shape `flow-node`; вид ноды в поле `kind`, данные — `props.extra` (JSON). Контекст для ИИ-нод собирается по входящим стрелкам (`extractNodeContext`).
- Оркестратор: `webbuild.ts` (сборка артефактов), `lecture.ts` (лекции), мост `webLLMAsk` (contracts → worker → broker → IPC `orch:askWebLLM` / `orch:webLLMResult`), реестры `agentTranscripts`, `webLLMRegistry`. Режимы: pipeline, actor-critic, council, ensemble, recursive.
- Хранение: `%APPDATA%/flow/` (`providers.json`, `settings.json`, `pdf/`, `anythingllm/`); память доски — `localStorage: flow-boardmem-<boardId>`; доски локально, опциональный realtime-синк через `sync-server/` (Cloudflare worker).
- Сборка: `npm run dev` / `npm run build` / `npm run dist` (установщик + Flow.exe).

### 0.1 Общие правила (действуют для всех задач)

1. **Обратная совместимость.** Существующие доски, память, ключи и настройки пользователей не должны ломаться. Любая смена формата данных — только через миграцию с бэкапом исходника в `%APPDATA%/flow/backup/`.
2. **Фиче-флаги.** Новое поведение — за флагами в `settings.json`. Инфраструктурные изменения (БД, логи) включены по умолчанию, поведенческие (реранкер, OCR, ленивые webview и т.п.) — с дефолтами, указанными в задаче. Мягкая деградация: если зависимость недоступна (нет модели, нет сети) — фича отключается, приложение продолжает работать.
3. **TypeScript strict.** Для всех новых структур данных — zod-схемы рядом с типами. Валидация всех IPC-пейлоадов — в main; renderer не считается доверенным.
4. **Безопасность.** Не ослаблять contextIsolation/sandbox. Секреты (ключи провайдеров) никогда не логировать и не отдавать в renderer в открытом виде.
5. **Сеть.** Никакой телеметрии. Исходящие запросы — только к настроенным провайдерам и к API, явно перечисленным в задачах.
6. **UI-строки — на русском**, в стиле существующего интерфейса.
7. **После каждой задачи:** typecheck чистый, `npm run build` проходит, ручной smoke: открыть старую доску, создать по ноде основных kind, запустить простой ран оркестратора, дропнуть PDF.
8. **Коммиты** небольшие, с номером задачи: `feat(T1.1): ...`, `fix(T2.2): ...`.

## Эпик 1 — Фундамент и надёжность

### T1.1 SQLite вместо localStorage для памяти доски

**Цель.** Убрать лимиты (~5–10 МБ) и хрупкость localStorage; получить единую локальную БД с полнотекстовым поиском как фундамент для T1.5, T2.4, T3.4, T4.1.

**Объём.**
- Подключить `better-sqlite3` в main-процессе. Учесть пересборку нативного модуля под Electron (electron-builder пересобирает нативные зависимости при `dist`; для dev — `electron-rebuild`). БД: `%APPDATA%/flow/flow.db`, режим WAL.
- Новый модуль `src/main/db.ts`: инициализация, миграции схемы БД (таблица `schema_migrations`, миграции — пронумерованные файлы/функции), подготовленные стейтменты, аккуратное закрытие при quit.
- Таблицы:
  - `board_memory(board_id, period_kind /*day|week|month*/, period_key, content, created_at, updated_at)` — выжимки;
  - `board_meta(board_id, key, value)` — флаги таймлайна и прочие пер-борд настройки;
  - FTS5-таблица `memory_fts` по контенту выжимок + триггеры синхронизации insert/update/delete.
- IPC + мост `window.flow`: `memory:list/get/upsert/delete/search`, `boardmeta:get/set`. Renderer полностью переводится с прямого localStorage на этот API (точки чтения/записи памяти и флагов таймлайна в `App.tsx`).
- Одноразовая миграция: при первом запуске новой версии renderer собирает все ключи `flow-boardmem-*` и флаги таймлайна → отправляет в main → main пишет в БД и сохраняет сырой дамп в `%APPDATA%/flow/backup/localStorage-<дата>.json` → только после подтверждения записи ключи в localStorage помечаются перенесёнными (не удалять, пометить префиксом/флагом).

**Критерии приёмки.**
- Память доски и флаги таймлайна работают через БД; после миграции старые выжимки видны в ноде «Память доски» и в панели таймлайна.
- Отсутствие/повреждение `flow.db` не роняет приложение: файл пересоздаётся, ошибка в логе, пользователю — уведомление.
- `memory:search` возвращает совпадения по подстроке/словам из выжимок разных досок.
- `npm run dist` собирает установщик, приложение стартует на чистой Windows-машине (нативный модуль корректно пересобран).

**Файлы.** `src/main/db.ts` (новый), `src/main/index.ts` (инициализация + IPC), `src/preload/index.ts`, `src/renderer/src/App.tsx`, `flowapi.d.ts`.

### T1.2 Версионирование схемы нод (zod + миграции)

**Цель.** `props.extra` — свободный JSON; эволюция форматов рано или поздно ломает старые доски. Нужны схемы и миграции по каждому `kind`.

**Объём.**
- Каталог `src/renderer/src/shapes/schemas/`: zod-схема на каждый `kind` + поле `extraVersion: number` внутри `extra` (отсутствие поля = версия 1).
- Реестр миграций: `migrateExtra(kind, extra): extra` — цепочка функций `vN → vN+1`, чистые, с юнит-тестами (vitest).
- При загрузке доски прогонять все `flow-node`: `safeParse` → при несоответствии применить миграции → при неудаче нода помечается «повреждена» (рендер: kind + свёрнутый raw JSON, данные не теряются, холст не падает).
- Дальнейшее правило (зафиксировать в §0.1-стиле в README схем): любое изменение формата `extra` = новая версия + миграция + тест.

**Критерии приёмки.**
- Старые доски открываются без визуальных отличий.
- Тестовая миграция (переименование поля в схеме `note`) проходит юнит-тест и корректно применяется при открытии доски со старым форматом.
- Нода с намеренно испорченным `extra` (невалидный JSON-объект) отображается как «повреждена», остальной холст работает.

**Файлы.** `src/renderer/src/shapes/schemas/*` (новые), `FlowNodeShapeUtil.tsx`, точка загрузки доски в `App.tsx`, тесты.

### T1.3 Шифрование ключей провайдеров (safeStorage)

**Цель.** `providers.json` хранит API-ключи в открытом виде. Зашифровать через ОС-хранилище.

**Объём.**
- Использовать Electron `safeStorage` (на Windows — DPAPI): `isEncryptionAvailable()` при старте.
- Формат файла v2: `{ "version": 2, "payload": "<base64(encryptString(JSON))>" }`. Чтение: v2 → расшифровка; plaintext (v1) → прочитать, сделать бэкап `providers.v1.bak.json`, перезаписать в v2.
- Если шифрование недоступно — остаёмся на plaintext, одноразовое предупреждение в настройках.
- Расшифрованные ключи живут только в main. В renderer (диалог настроек) ключи отдаются маской `sk-…последние4`; кнопка «показать» запрашивает полное значение точечно через IPC.
- Санитайзер логов: значения ключей никогда не попадают в логи (см. T1.6).

**Критерии приёмки.**
- После первого запуска файл на диске не содержит ключей в открытом виде; все провайдеры работают как раньше.
- Маска в UI корректна; «показать» отображает ключ; редактирование и сохранение ключа работает.
- Откат: при повреждении v2-файла приложение сообщает об ошибке и предлагает ввести ключи заново (не падает).

**Файлы.** модуль чтения/записи providers в `src/main/`, диалог настроек в renderer, `preload`.

### T1.5 Персистентные раны оркестратора (event log + resume)

**Цель.** Ран, прерванный крашем или закрытием приложения, должен продолжаться, а не начинаться заново. Совместимо с уже принятой идеологией event sourcing / deterministic replay из мастер-плана.

**Объём.**
- Таблицы (T1.1): `runs(id, board_id, mode, status, created_at, finished_at, meta)`, `run_events(run_id, seq, type, node_id, payload_json, ts)`.
- Брокер пишет события: `run_created`, `node_started`, `node_output`, `node_failed`, `hitl_wait`, `hitl_resolved`, `run_done`, `run_failed`. Payload’ы — компактные (большие выходы можно класть целиком: БД локальная).
- Статусы: при старте приложения раны в состоянии «выполнялся» переводятся в `interrupted`.
- «Продолжить ран»: восстановление состояния из событий — завершённые узлы не перевыполняются (их выходы берутся из `node_output`), выполнение продолжается с первого незавершённого узла. Идемпотентность повторов усиливается кэшем T3.4, но не зависит от него.
- UI: список ранов доски (статус, время, режим), действия «Продолжить» / «Открыть трейс».

**Критерии приёмки.**
- Принудительное завершение приложения посреди многоузлового рана → после запуска ран виден как `interrupted`; «Продолжить» доводит его до конца, завершённые узлы не вызывают модель повторно.
- Трейс восстановленного рана полный (старые + новые события в одном списке).
- HITL-пауза переживает перезапуск приложения.

**Файлы.** брокер/воркер оркестратора в `src/main/`, `db.ts`, UI трейса/ранов в renderer.

## Эпик 2 — RAG и память

### T2.1 Гибридный поиск + реранкер (локальный PDF-RAG)

**Цель.** Чисто векторный поиск промахивается на точных терминах (аббревиатуры, числа, названия методов). Гибрид BM25 + эмбеддинги + опциональный локальный реранкер.

**Объём.**
- Лексический индекс по чанкам каждого `pdf_id`: `minisearch` (BM25-подобный скоринг) с персистом рядом с данными PDF (`%APPDATA%/flow/pdf/<id>/`), либо `wink-bm25-text-search` — на выбор исполнителя, критерий: сериализация индекса и русская/английская токенизация.
- Retrieval-конвейер: top-40 лексических + top-40 векторных → слияние Reciprocal Rank Fusion → top-N (настройка, по умолчанию 8).
- Опциональный реранкер: `onnxruntime-node` + ONNX-модель `bge-reranker-v2-m3`. Модель не кладётся в репозиторий: кнопка «Скачать модель реранкера» в настройках, хранение в `%APPDATA%/flow/models/`, реранк top-40 → top-8.
- Флаги: `rag.hybrid` (по умолчанию вкл), `rag.reranker` (по умолчанию выкл; сереет, пока модель не скачана). Без модели/при ошибке — мягкий фолбэк на гибрид без реранка; без гибрида — текущее поведение.
- Замер и лог времени этапов retrieval (скоуп `rag`).

**Критерии приёмки.**
- Тест-кейс: PDF с редким точным термином, который эмбеддинги не находят в топе, — гибрид выводит нужный чанк в top-N.
- Выключение флагов возвращает прежнее поведение.
- Реранкер работает офлайн после скачивания модели; его отсутствие не ломает Q&A.

**Файлы.** модуль PDF-RAG в `src/main/`, настройки, `docs/CHANGELOG-improvements.md`.

### T2.2 Цитаты с якорями страниц

**Цель.** Ответ по PDF должен ссылаться на страницы; клик по цитате — открывать PDF-ноду в нужном месте. Ключевая фича для ресёрч-сценария.

**Объём.**
- При индексации каждый чанк получает метаданные: `page` (обязательно), `charStart/charEnd` в тексте страницы (по возможности). Существующие индексы: фоновая переиндексация при первом Q&A или кнопка «Переиндексировать» на ноде.
- Промпт Q&A: источники нумеруются `[1]…[n]` с указанием страниц; инструкция модели ставить маркеры `[n]` после утверждений.
- Парсинг маркеров в ответе → рендер кликабельных ссылок в answer-ноде/панели ответа. Клик: сфокусировать/открыть соответствующую pdf-ноду → перейти на страницу → подсветить текст чанка (поиск подстроки в текстовом слое; если не нашлась — просто перейти на страницу).
- Если модель не проставила маркеры — в конце ответа всё равно выводится блок «Источники: с. X, Y, Z» по использованным чанкам.

**Критерии приёмки.**
- Ответ на вопрос по PDF содержит хотя бы одну кликабельную цитату; переход открывает нужную страницу с подсветкой.
- Блок «Источники» появляется всегда, когда retrieval вернул чанки.
- Старые PDF без метаданных страниц работают после переиндексации.

**Файлы.** PDF-RAG (индексация, промпт), просмотрщик PDF-ноды, рендер ответа.

### T2.3 Структурный чанкинг + OCR

**Цель.** Чанки по границам смысла, а не по числу символов; поддержка скан-PDF.

**Объём.**
- Чанкинг с учётом структуры: эвристика заголовков по `textContent` items из pdf.js (размер шрифта/жирность/позиция) → резать по секциям, крупные секции — по абзацам с перекрытием; не рвать предложения.
- Таблицы: эвристика табличных блоков (выравнивание по колонкам) → отдельные чанки с пометкой `type: table` (в промпте помечать как таблицу).
- OCR: если на странице почти нет текстового слоя — `tesseract.js` (языки rus+eng, traineddata скачиваются один раз в `%APPDATA%/flow/models/ocr/`). Флаг `rag.ocr` (по умолчанию выкл), прогресс OCR — в UI PDF-ноды.
- Кнопка «Переиндексировать» на PDF-ноде (используется и в T2.2).

**Критерии приёмки.**
- На типовой научной статье чанки совпадают с секциями (проверка глазами по 2–3 статьям), середина предложения между чанками не рвётся.
- Скан-PDF при включённом OCR становится доступен для Q&A; при выключенном — честное сообщение «текстовый слой отсутствует».
- OCR не блокирует UI (в worker/асинхронно), прогресс виден.

**Файлы.** PDF-RAG (чанкер), PDF-нода (UI), настройки.

### T2.4 Память доски через retrieval (вместо «вся память в контекст»)

**Цель.** На доске возрастом в месяцы полная память взрывает бюджет токенов. Подтягивать релевантное, а не всё.

**Объём.**
- Эмбеддинг каждой выжимки при создании/изменении (та же модель эмбеддингов, что в PDF-RAG). Таблица `memory_embeddings(board_id, memory_rowid, vector BLOB)`; для старых выжимок — фоновая доиндексация.
- Новая сборка контекста «Память → ИИ/оркестратор»: последние N дней целиком (по умолчанию 7) + top-k релевантных запросу старых дневных выжимок (по умолчанию 6, косинусная близость) + недельные/месячные сводки, всё в пределах бюджета `memory.contextBudget` (токены, по умолчанию 4000; счётчик — `gpt-tokenizer`, приближённо).
- Релевантность считается к тексту текущего запроса пользователя/узла; если запроса нет (превью в ноде «Память доски») — просто последние + сводки.
- Авто-выжимки дня/недели/месяца выполняются моделью роли `summarize` (T3.3), по умолчанию локальной.
- Флаг `memory.retrieval` (по умолчанию вкл после доиндексации); фолбэк — текущее поведение.

**Критерии приёмки.**
- Синтетическая доска с 60+ днями: контекст памяти в собранном промпте ≤ бюджета; тематический вопрос вытягивает релевантную старую выжимку (проверить в инспекторе контекста T3.1).
- Старые доски без эмбеддингов работают сразу (фолбэк) и мигрируют в фоне.

**Файлы.** модуль памяти (main + renderer), `db.ts`, сборка контекста, настройки.

### T3.4 Кэш LLM-вызовов

### T3.5 Устойчивость веб-нод: структурные транскрипты, health, детект разлогина

**Цель.** `innerText` хрупок к вёрстке провайдеров и не различает роли. Round-robin не должен слать задачи мёртвым нодам.

**Объём.**

- Слом селекторов (симулировать) не роняет снятие текста — работает фолбэк.

**Файлы.** `src/main/webllm/*`, преролл-скрипты webview, `webbuild.ts`, UI бейджей ноды.

### T4.3 Масштаб таймлайна: плотность, «перейти к дате», сворачивание прошедшего

**Цель.** Фиксированные 1700 px/день неудобны на дистанции месяцев.

**Важное ограничение.** Ось Y — абсолютные координаты холста, дата ноды следует из её `y`. Любая смена плотности обязана согласованно пересчитать и разметку, и координаты нод — иначе ноды «переедут» на другие даты.

**Объём.**
- Вынести всю математику дат в чистый модуль `timelineMap`: `dateToY(config)`, `yToDate(config)`, где config = `{pxPerDay, collapsedPeriods[]}` per board (хранится в `board_meta`). Вся отрисовка разметки, логика «день в центре экрана», выжимок и сжатия пустых периодов переводится на этот модуль (сейчас константа 1700 разбросана по коду — собрать в одном месте).
- Режимы плотности per board: «День» (1700 px/день, как сейчас), «Неделя» (~400 px/день), «Месяц» (~120 px/день). Переключение: batch-пересчёт `y` всех нод через `yToDate(старый) → dateToY(новый)` одним undo-шагом; подписи разметки адаптируются (в компактных режимах — подписи недель/месяцев).
- «Перейти к дате»: пикер в меню доски → скролл камеры к `dateToY(дата)`.
- Сворачивание прошедших недель: свёрнутый период рисуется тонкой полосой с бейджем «Неделя N (свёрнута)»; реализовать через `collapsedPeriods` в `timelineMap` с тем же batch-пересчётом координат нод (свёрнутые ноды скрываются/показываются стеком-иконкой; разворот возвращает всё на место). Существующее «удаление пустого периода» переиспользует тот же механизм, но остаётся отдельной необратимой операцией.

**Критерии приёмки.**
- Переключение плотности туда-обратно возвращает каждую ноду в исходную дневную полосу (инвариант date(node) сохраняется), undo работает.
- «Перейти к дате» попадает в нужную полосу во всех режимах.
- Свёрнутая неделя: ноды скрыты, ниже лежащие полосы поднялись; разворот полностью обратим; выжимки/память продолжают считаться по настоящим датам.

**Файлы.** `timelineMap` (новый модуль), таймлайн-слой в `App.tsx`, `board_meta`.

### T4.5 Автолейаут кластеров webbuild

**Цель.** Пачки артефактов webbuild должны читаться без ручного растаскивания.

**Объём.**
- Раскладка через `dagre` (при потребности в ортогональных рёбрах/группах — `elkjs`, решение за исполнителем): после генерации пачки — layered-раскладка нод внутри каждой группы (направление слева-направо по стрелкам), группы — сеткой справа от ресёрч-зоны; учитывать фактические размеры нод.
- Плавная анимация перелёта нод на позиции; один undo-шаг.
- Команда «Разложить выделенное» в контекстном меню — тот же алгоритм для произвольного выделения нод со стрелками.

**Критерии приёмки.**
- Результат полного webbuild: ноды не пересекаются, стрелки преимущественно слева-направо, группы визуально разделены.
- «Разложить выделенное» корректно работает на ручном графе из 6+ нод; undo возвращает позиции.

**Файлы.** `webbuild.ts` (координаты), модуль лейаута (renderer), контекстное меню.

## Эпик 5 — Новые возможности

### T5.1 Библиография: Crossref / Semantic Scholar / arXiv + BibTeX/Zotero

**Цель.** Ресёрч-фаза сильнее и с нормальной библиографией — прямая ценность для обзоров литературы.

**Объём.**
- Единый интерфейс `PaperSource { search(query, filters): PaperMeta[] }`; `PaperMeta = {title, authors, year, doi?, abstract?, venue?, url, source}`. Адаптеры: OpenAlex (существующий — привести к интерфейсу), Crossref (передавать `mailto` из настроек — вежливый пул), Semantic Scholar (API-ключ опционален, уважать rate-limit, ретраи), arXiv (Atom XML).
- Слияние результатов с дедупликацией по DOI, при его отсутствии — по нормализованному заголовку+году; в ресёрч-фазе источники помечаются, откуда пришли.
- Нода «база знаний»: вкладка «Литература» — список найденных работ с чекбоксами; экспорт выбранного в `.bib` (генерация BibTeX; допустимо через `@citation-js/core`).
- Опционально: пуш выбранных работ в Zotero Web API (userID + ключ в настройках, выбор коллекции); за флагом, при ошибке — понятное сообщение.

**Критерии приёмки.**
- Ресёрч по тестовой теме возвращает работы минимум из двух источников без дублей (проверить на теме с известным пересечением arXiv/Crossref).
- `.bib` импортируется в Zotero/JabRef без ошибок; спецсимволы в заголовках экранированы.
- Отключённый Semantic Scholar (нет ключа, лимит) не ломает ресёрч — остальные источники работают.

**Файлы.** `src/main/research/sources/*` (новые адаптеры), ресёрч-фаза оркестратора, нода базы знаний, настройки.

### T5.2 Планировщик (автоматизации)

**Цель.** Регулярные задачи — например, ночная выжимка дня — без участия пользователя.

**Объём.**
- `settings.json`: `automations: [{id, schedule: "daily@HH:MM", action: "dailyDigest" | "boardDigest", boardId, enabled, catchUp: bool}]`.
- Исполнение в main по таймеру при запущенном приложении; при `catchUp` — догоняющий запуск пропущенной задачи при старте. Действия используют роль `summarize` (T3.3) и существующую логику выжимок.
- Лог запусков в БД (`automation_runs`), уведомление о результате (успех/ошибка).
- UI «Автоматизации» в настройках: список, вкл/выкл, время, доска, история запусков.

**Критерии приёмки.**
- Ежедневная выжимка выбранной доски создаётся в назначенное время локальной моделью; ручная выжимка при этом не дублируется (идемпотентность по дню).
- Пропущенный запуск при `catchUp=true` отрабатывает при следующем старте один раз.

**Файлы.** планировщик в main, настройки UI, `db.ts`.

### T5.3 Браузер-агент нода (Playwright) — низкий приоритет, объём согласовать отдельно

**Рамка.** Отдельная нода `browser-agent`: main управляет Playwright-Chromium (headful, отдельное окно), агент-цикл «цель → навигация/клик/извлечение», результаты — markdown-заметками на доску. Обязательно: подтверждение пользователем любых действий отправки форм; без логинов в аккаунты; строго за флагом. **В рамках этого пакета** — только прототип на один сценарий «пройти по списку ссылок из ноды и собрать заголовки+абстракты», после демо — решение о развитии.

## Приложение А — как работать с этим ТЗ в Claude Code

1. Положить файл в репозиторий как `docs/TZ-improvements.md`. В корневой `CLAUDE.md` добавить ссылку на него и продублировать правила §0.1 (кратко).
2. Одна задача — одна сессия/ветка. Стартовый промпт:

```
Прочитай docs/TZ-improvements.md (раздел 0 целиком и задачу TX.Y).
Сначала составь план изменений: файлы, шаги, риски, что уточнить. Код пока не пиши.
После моего подтверждения реализуй задачу, прогони typecheck и npm run build,
пройди «Критерии приёмки» и smoke-чек-лист из §0.1 п.7, покажи результат.
```

3. Вопросы, помеченные в задачах как «на выбор исполнителя», Claude Code решает сам и фиксирует выбор в CHANGELOG; всё, что помечено «согласовать с владельцем», — останавливается и спрашивает.
4. После приёмки задачи — коммит `feat(TX.Y): …` и запись в `docs/CHANGELOG-improvements.md`.
