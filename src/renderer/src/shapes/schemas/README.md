# Схемы и версионирование `props.extra` (T1.2)

`flow-node` хранит данные в `props.extra` — свободном JSON. Чтобы эволюция форматов не
ломала старые доски, здесь для каждого `kind` заведены zod-схема, целевая версия и
цепочка миграций.

## Железное правило

**Любое изменение формата `extra` = НОВАЯ версия + миграция `vN→vN+1` + юнит-тест.**

Нельзя менять смысл/имя поля «на месте»: старые доски встретят новый код со старыми
данными. Вместо этого:

1. Увеличь `CURRENT_EXTRA_VERSION[kind]` на 1 (по умолчанию версия = 1).
2. Добавь чистую функцию миграции в `MIGRATIONS[kind][старая_версия]` — она превращает
   `extra` версии N в версию N+1 (например, переименовывает поле).
3. При необходимости обнови zod-схему в `EXTRA_SCHEMAS[kind]` под новый формат
   (схемы **пермиссивные**: `catchall(unknown)` сохраняет незнакомые поля — не удаляй
   данные пользователя).
4. Добавь юнит-тест в `__tests__/migrations.test.ts`.

## Как это работает при загрузке доски

`migrateBoardExtras(editor)` (в `FlowNodeShapeUtil.tsx`) прогоняет каждую `flow-node`
через `parseAndMigrateExtra(kind, raw)`:

- невалидный JSON или не-объект → нода помечается **повреждённой** (`markCorrupt`);
  рендерится заглушка `CorruptedBody` (kind + свёрнутый сырой JSON), данные не теряются,
  холст не падает;
- иначе применяются миграции до текущей версии; если `extra` изменился — он однократно
  перезаписывается в сторе (тихо, без засорения undo).

Схемы **не ужесточают** существующие поля без миграции — цель модуля защитить обратную
совместимость, а не отвергать валидные старые данные.

## Пример (реализован для `note`)

`note` — версия 2. Миграция `v1→v2` переименовывает устаревшее поле `pinned` → `favorite`.
Реальные заметки хранят текст в `props.title`/`props.body`, `extra` обычно пуст, поэтому
для них миграция безопасна (просто проставит `extraVersion: 2`). См. тест
`__tests__/migrations.test.ts`.
